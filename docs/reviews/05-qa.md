## Bugs found and fixed

### B1 [HIGH] — False success toast when YouTube upload fails
**Found in**: Test 5
**Symptom**: When YouTube OAuth is not configured, `POST /api/long-form` catches the error internally and returns HTTP 200 with `status: "failed"`. The dialog's success handler saw `r.ok === true` and showed "Long-form scheduled on YouTube", lying to the user.
**Root cause**: `src/components/new-long-form-dialog.tsx:57-64` only checked `r.ok`, not `d.item.status`.
**Fix**: Added a branch that checks `d.item.status === "failed"` and shows an error toast instead: "YouTube upload failed — check OAuth config in Settings. Saved as failed."
**File**: `src/components/new-long-form-dialog.tsx`
**Verified**: re-ran test 5, toast now correctly shows the failure message.

## Bugs found and deferred

### B2 [MEDIUM] — Shorts generation creates duplicates on re-run
**Found in**: Test 7 (when run twice)
**Symptom**: Clicking "Generate shorts" on a long-form that already has shorts creates 6 more shorts (12 total). The scheduler then tries to schedule 12 shorts, pushing posts far into the future.
**Root cause**: `src/lib/shorts-pipeline.ts` has no dedup check.
**Fix prescribed**: Add a confirmation dialog ("This video already has N shorts. Replace them?") and delete existing shorts before regenerating.
**Tracked as**: T8 in `01-plan-design-review.md`

### B3 [LOW] — Chart empty states show demo data
**Found in**: visual inspection (no test — empty DB scenario)
**Symptom**: When the DB has no scheduled posts, charts render synthetic sin-wave demo data instead of an empty state.
**Root cause**: `conversation-volume-chart.tsx`, `csat-responses-chart.tsx`, `first-reply-time-chart.tsx` all fall back to `buildLastNDays()` with random demo values.
**Fix prescribed**: Show an empty-state card when the API returns zero rows.
**Tracked as**: T2 in `01-plan-design-review.md`
